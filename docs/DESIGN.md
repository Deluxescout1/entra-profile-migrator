# EntraProfileMigrator — Design

An in-house replacement for ForensiT User Profile Wizard, scoped to the migration we actually do:
**on-prem AD (domain-joined) → Entra ID (cloud-only) joined**, deployable via NinjaOne / Rewst,
so the end user only signs in with their Microsoft account.

---

## 1. The problem

When a Windows device leaves the domain and joins Entra ID, the user's new identity gets a brand
new local SID. Windows keys profiles by SID, so by default the user logs in to an empty profile
and loses Documents, Desktop, Outlook cache (OST), browser profiles, app configs, mapped settings —
everything. We need to attach the **existing** profile to the **new** identity instead.

Two things are keyed by SID and must be made consistent for the new identity:
1. **The filesystem** — `C:\Users\<name>` and its NTFS ACLs reference the old domain SID.
2. **The registry** — `HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\ProfileList\<SID>`
   maps a SID → `ProfileImagePath` (the folder). The per-user hive `NTUSER.DAT` also carries
   ACLs and some SID-referencing values.

---

## 2. The hard constraint: Entra SIDs are minted at first sign-in

For a domain-to-domain move you can resolve the target SID up front. For **Entra you cannot** —
the user's *local* SID is generated by Windows the first time that Entra account signs in on that
specific device. It is not derivable from the Object ID / UPN ahead of time in a reliable way.

### Decision: post-first-login re-permission (v1)
We require the target user to sign in **once** with their Entra account, which mints the SID and a
throwaway profile. We then:
- discover the new SID from `ProfileList`,
- re-point it at the old profile,
- discard the throwaway profile,
- prompt one reboot.

This is the same reality the ForensiT *freeware* path lives with. It is robust and predictable.

### Stretch goal (do NOT build in v1): zero-extra-login pre-staging
Pre-create the `ProfileList` mapping before first login so the user lands in the migrated profile
on their very first Entra sign-in. This requires obtaining/forcing the local SID before an
interactive login (CloudAP internals). Treat as research; gate behind a feature flag; never make
v1 depend on it.

---

## 3. Migration mechanics

### 3a. Re-ACL the filesystem (SID substitution, not just "add access")
Use the documented `icacls` save/restore-with-substitute workflow on the live tree. It both
rewrites old→new SID across every ACE **and** the saved file doubles as the rollback artifact:

```
icacls "C:\Users\<name>" /save  "<backup>\fs-acl.txt" /T /C /Q
icacls "C:\Users\<name>" /restore "<backup>\fs-acl.txt" /substitute <OldSID> <NewSID> /T /C /Q
icacls "C:\Users\<name>" /setowner <NewSID> /T /C /Q
```

Then a sanity grant so the new SID has explicit FullControl on the profile root.

### 3b. Rewrite the NTUSER.DAT hive ownership/ACLs
The target must not be logged on (hive not loaded). Then:

```
reg load HKU\EPM_TMP "C:\Users\<name>\NTUSER.DAT"
# grant <NewSID> FullControl + set owner recursively on Registry::HKEY_USERS\EPM_TMP
reg unload HKU\EPM_TMP
```

`icacls` cannot do `/substitute` on the registry, so hive ACLs are handled with the .NET
`RegistrySecurity` APIs (recursive walk). Do the same for `UsrClass.dat`
(`...\AppData\Local\Microsoft\Windows\UsrClass.dat`). If a full *value-level* SID rewrite inside
the hive is ever needed, `SetACL.exe` is an optional, feature-flagged helper — not a hard dep.

### 3c. Rewrite the ProfileList mapping
Given old SID `S_OLD` (points at the real data folder) and new SID `S_NEW` (created at first
login, points at a sparse `C:\Users\<name>.<TENANT>`):

1. Back up: `reg export "HKLM\...\ProfileList" "<backup>\profilelist.reg"`.
2. Under `ProfileList\<S_NEW>`: set `ProfileImagePath` to the **old** data folder; reset `State`,
   `RefCount`, `Flags` to clean values; fix the binary `Sid` value if present.
3. Remove (after backing up) `ProfileList\<S_OLD>`.
4. Delete or rename the sparse `C:\Users\<name>.<TENANT>` folder created by the throwaway login.

**Folder naming decision:** keep the original folder path (`C:\Users\<name>`). Do *not* rename it —
renaming risks breaking absolute paths baked into the profile/registry/apps. Document any tenant
suffix collision handling.

### 3d. Order of operations (atomic-ish)
prereqs → backup → FS re-ACL → hive re-ACL → ProfileList rewrite → cleanup throwaway → mark
migrated → reboot prompt. Any failure after backup writes a `ROLLBACK.txt` next to the backup with
exact restore commands.

## 4. Source/target identification
- Enumerate `ProfileList`; resolve each SID via `SecurityIdentifier.Translate([NTAccount])`.
- **Domain (source) candidates:** `S-1-5-21-...` that resolve to the old domain or no longer
  resolve (DC gone), excluding well-known/system SIDs and the local machine.
- **Entra (target):** SID resolves to `AzureAD\user@domain`, or folder name carries the tenant
  suffix. `Resolve-TargetSid` matches a target UPN to its freshly-minted SID.
- **Mapping:** optional CSV (`old sAMAccountName/UPN` → `new Entra UPN`) for bulk/unattended runs,
  mirroring ForensiT's lookup file. Single-machine interactive runs can auto-pair when there's one
  obvious domain profile and one new Entra profile.
- Skip: already-migrated (manifest/marker present), special/service profiles, FSLogix/roaming
  containers.

---

## 5. Edge cases to handle (track as issues)
- **OneDrive Known Folder Move** already syncing Desktop/Documents/Pictures → detect; data-loss
  risk is lower but app/Outlook/browser state still needs migration. Report it.
- Outlook OST rebuild (acceptable; cached mode re-syncs) vs. PST files (must be preserved).
- BitLocker-encrypted volume (fine, but log it).
- Roaming/mandatory profiles, temp profiles (`State` bit set), `.bak` ProfileList keys.
- Profile in use / files locked by AV → must run from SYSTEM with target logged off.
- Multiple domain profiles on one shared machine (loop; map each).
- Very large profiles (in-place is fast; copy mode is the slow/safe alternative — see §6).
- Re-run after partial failure (idempotency + rollback).

---

## 6. In-place vs copy
Default: **in-place** (fast, no extra disk) — what ForensiT does by default. Provide an opt-in
`-Copy` mode that robocopies the profile to a fresh folder and re-ACLs the copy, leaving the
original untouched, for high-value machines. Copy mode = safer rollback, 2× disk, slower.

---

## 7. Deployment model
- Runs as **SYSTEM** via NinjaOne script policy (`deploy/NinjaOne-Run-Migration.ps1`) or a Rewst
  workflow that pushes params (mapping CSV path, mode, reboot policy).
- Entra **join itself is out of scope for the core** (assume Autopilot or a provisioning package
  pushed separately). Provide `Test-MigrationPrerequisite` to verify join state via
  `dsregcmd /status` (AzureAdJoined : YES).
- Exit codes: `0` success, `10` dry-run-only, `20` prereq fail, `30` no migratable profile,
  `40` target SID not found (user hasn't signed in yet), `50` migration error (rolled back),
  `60` migration error (manual rollback needed). Keep this table in sync with the code.

---

## 8. Out of scope (v1)
Entra join automation, Autopilot enrollment, BitLocker handling beyond logging, app reinstall,
mailbox migration, file-share/printer re-mapping, hybrid-join scenarios (this is cloud-only join).

---

## 9. Build checklist (order)
- [ ] Module skeleton loads; manifest exports public funcs.
- [ ] `Get-ProfileListEntry` (private) — read + classify all ProfileList entries. **Read-only.**
- [ ] `Get-MigratableProfile` (public) — classified source/target list. **Read-only.**
- [ ] `Resolve-TargetSid` (private) — UPN → minted SID. **Read-only.**
- [ ] `Test-MigrationPrerequisite` (public) — join state, admin, target logged-off, disk, backups
      writable. **Read-only.** Prove all of the above in a VM before writing anything mutating.
- [ ] `New-MigrationBackup` (private) — ProfileList export + FS ACL save + manifest.
- [ ] `Set-ProfileSidOwnership` (private) — icacls substitute/setowner/grant.
- [ ] `Update-RegistryHiveSid` (private) — load hive, re-ACL, unload.
- [ ] `Update-ProfileListMapping` (private) — rewrite mapping, remove old key, cleanup throwaway.
- [ ] `Invoke-ProfileMigration` (public) — orchestrator, WhatIf default, rollback on failure.
- [ ] Pester tests per function; integration test harness for the VM.
- [ ] NinjaOne wrapper + Rewst notes.
- [ ] Rollback command (`Restore-MigrationBackup`).
